## 一、变更确认机制（最高优先级）

> **核心原则：先确认，再动手。大改动必须先对齐方案，杜绝擅自推倒重来。**

### 1.1 必须事先确认的操作

以下行为**禁止未确认直接执行**，须先向项目负责人说明意图、范围、影响，获得明确同意后方可进行：

| 类别 | 典型场景 |
|------|----------|
| **架构级变更** | 引入 / 替换技术栈；调整项目目录结构；拆分或合并服务模块 |
| **破坏性变更** | 修改公开 API 契约；变更数据库 Schema（删表、改列、改约束）；调整序列化 / 反序列化格式；修改认证与授权模型 |
| **大规模重构** | 跨 5 个以上文件的命名 / 结构重构；提取或消除公共模块 |
| **依赖变更** | 新增第三方依赖；升级主版本号跨度 ≥ 1 的依赖；替换已有依赖 |
| **配置变更** | 修改环境变量契约；调整构建 / 部署流程配置；变更 CI/CD 流水线 |
| **删除操作** | 删除 3 个以上文件；删除被其他模块引用的导出；删除已有功能代码 |

### 1.2 确认方式

1. 口头或书面描述变更目标、范围、影响面、回滚方案。
2. 获得确认后，创建对应的 Git 分支开始实施。
3. 实施过程中若范围发生扩大，须再次确认。

### 1.3 可直接执行的操作

以下操作无需事先确认，但仍须遵循本文件其余规范：

- 单文件内 Bug 修复
- 补充单元测试
- 更新文档（不含架构类文档）
- 代码格式化与注释补充
- 非破坏性依赖的 patch 版本升级

## 二、Git 版本记录与备份规范

> **Git 是第一道安全网：每次提交都是可回溯的存档点，每次推送都是异地备份。**

### 2.1 仓库管理

- 项目根目录下必须维护 `.gitignore`，排除构建产物、依赖目录、环境变量文件、IDE 个人配置、系统生成文件。
- `main` 分支受保护，禁止直接推送，仅通过 PR 合入。
- 远程仓库作为权威备份源，本地提交应在每个工作日结束前推送至远程。

### 2.2 分支策略

| 分支 | 用途 | 命名示例 |
|------|------|----------|
| `main` | 生产就绪代码，只接受经过审查的 PR 合入 | — |
| `develop` | 日常集成，feature / fix 分支的合入目标 | — |
| `feature/<slug>` | 新功能开发 | `feature/invoice-parser` |
| `fix/<slug>` | 常规缺陷修复 | `fix/currency-rounding` |
| `hotfix/<slug>` | 生产紧急修复，从 `main` 分出 | `hotfix/api-timeout` |
| `release/<version>` | 发布候选，冻结功能仅做修复 | `release/v1.2.0` |

### 2.3 Commit 规范

严格遵循 **Conventional Commits** 格式：

```
<type>(<scope>): <subject>
```

- `subject` 使用中文或英文均可，但同一项目内保持一致。
- 单次提交只做一件事：一个逻辑变更一个 commit，拒绝"杂烩式提交"。
- `scope` 使用模块 / 服务目录名。

**type 取值表：**

| type | 用途 | 示例 |
|------|------|------|
| `feat` | 新增功能 | `feat(order): 添加订单导出 Excel 功能` |
| `fix` | 修复缺陷 | `fix(exchange): 修复日元汇率精度丢失` |
| `docs` | 文档变更 | `docs(readme): 更新本地启动说明` |
| `style` | 格式（不影响逻辑） | `style: 统一缩进为 2 空格` |
| `refactor` | 重构（不改变外部行为） | `refactor(auth): 提取 Token 验证中间件` |
| `perf` | 性能优化 | `perf(query): 为 trade_orders 表添加复合索引` |
| `test` | 测试 | `test(customs): 补充报关单校验单测` |
| `chore` | 构建 / CI / 依赖 | `chore(deps): 升级 axios 至 1.6.0` |
| `revert` | 回退提交 | `revert: 回退 feat(invoice): 添加 OCR 识别` |

### 2.4 备份纪律

- **每日推送**：工作日结束前将本地提交推送至远程仓库。
- **功能分支推送**：功能分支开发超过半天或离开工位前，推送至远程（即使未完成）。
- **标签打点**：每个发布版本必须打 annotated tag：`git tag -a v1.2.0 -m "发布说明"`。
- **仓库备份**：远程仓库（GitHub / GitLab / Gitee 等）作为主备份，建议定期镜像到第二平台。
- **禁止操作**：禁止 `git push --force` 到共享分支；禁止 `git reset --hard` 已推送的提交。

### 2.5 Git 操作安全规则

- 执行 `git push --force` / `git reset --hard` / `git clean -fdx` / `git rebase` 等高风险命令前，必须确认目标分支非共享分支。
- 合入前必须确保目标分支与本地无冲突，冲突需人工逐行解决，禁止使用 `--theirs` 或 `--ours` 一键覆盖。
- Squash / Rebase 仅限本地未推送的提交。

## 四、编码规范

### 4.1 通用原则

| 原则 | 说明 |
|------|------|
| **自解释命名** | 命名清晰到不需要注释解释"它是什么" |
| **单一职责** | 一个模块 / 函数只做一件事，边界清晰 |
| **防御性编程** | 外部输入必须校验；异常路径必须显式处理；拒绝静默失败 |
| **不可变优先** | 默认 const / readonly，可变状态限制作用域并文档化 |
| **最小依赖** | 非必要不引入；引入前评估体积、活跃度、许可证兼容性 |
| **显式优于隐式** | 拒绝魔法值、隐式类型转换、隐式全局状态 |

### 4.2 格式约定

- 编码：**UTF-8**，行尾：**LF**
- 缩进：TypeScript / 前端用 **2 空格**；Python / Go 用 **4 空格**
- 行宽上限：代码 **120 字符**，文档 **80 字符**
- 文件末尾保留一个空行；禁止行尾空白
- 魔法数字 / 字符串一律提取为命名常量（`0`、`1`、`-1`、`""` 等通用哨兵除外）

### 4.3 命名规范

| 实体 | 规范 | 示例 |
|------|------|------|
| 文件 / 目录 | kebab-case | `exchange-rate.ts`, `order-service/` |
| 类 / 接口 / 类型 | PascalCase | `TradeDocument`, `ICustomsInfo` |
| 函数 / 方法 / 变量 | camelCase | `calculateTariff()`, `shipmentId` |
| 常量 | UPPER_SNAKE_CASE | `MAX_RETRY`, `DEFAULT_CURRENCY` |
| 数据库表 | snake_case 复数 | `trade_orders`, `customs_declarations` |
| 数据库列 | snake_case | `created_at`, `buyer_company_name` |

### 4.4 注释规则

- 注释回答 **Why**，不重复代码的 **What**。
- 公开 API / 导出函数必须有 JSDoc / docstring。
- 临时方案 (workaround / hack) 必须标记 `// TODO:` 或 `// FIXME:` 并注明原因与日期。
- 禁止提交被注释掉的代码块 —— 依赖 Git 历史回溯。

## 五、测试规范

| 层级 | 最低覆盖 | 说明 |
|------|----------|------|
| 单元测试 | ≥ 80% | 核心业务逻辑为强制项；纯样板代码 (getter/setter/透传) 可豁免 |
| 集成测试 | 关键路径全覆盖 | API 契约、数据库交互、外部服务适配器 |
| E2E | 核心用户旅程 | 限关键流程，保持 ≤ 20 个场景避免维护债 |

- 测试命名：`<源文件>.test.<ext>` 或 `<源文件>.spec.<ext>`
- 测试文件与源文件同目录或镜像目录结构
- 测试须可独立运行、可重复、无状态共享
- 未通过测试的代码禁止合入主干

## 六、安全底线

> 外贸系统涉及商业机密与跨境合规，安全不可妥协。

- **禁止**在仓库中提交密钥、Token、证书、密码。一律使用环境变量或密钥管理服务。
- 用户输入一律视为不可信，后端必须独立校验，不可依赖前端校验。
- SQL 必须参数化，禁止字符串拼接。
- 敏感数据落库加密存储；传输层强制 TLS 1.2+。
- 日志中禁止输出明文密码、Token、个人身份信息。
- 认证 / 授权逻辑后端独立实现，前端不得作为唯一校验点。
- 定期扫描依赖漏洞，中高危漏洞必须在 1 个迭代内修复。
- **不受理、不实现**任何规避贸易管制、制裁法规或法律限制的功能需求。

---

## 七、性能基线

- 对外部服务调用默认超时 ≤ 30s，必须有重试（指数退避）与熔断。
- 面向用户的接口 P95 响应 ≤ 500ms（不含外部依赖耗时）。
- 数据库查询必须走索引；禁止在循环内执行 SQL 查询。
- 大数据量操作须分页或流式处理，禁止一次性全量加载到内存。
- 耗时任务走消息队列异步解耦，禁止阻塞请求-响应线程。

---

## 八、国际化（i18n）

- 所有面向用户的字符串抽取为翻译键，禁止硬编码中文 / 英文。
- 翻译文件集中管理，按语言分目录。
- 日期、货币、数字格式化使用标准库（`Intl`、`ICU`），不自行实现。
- 时间统一 UTC 存储，展示层按用户时区转换。

## 九、AI 辅助开发规则

- AI 生成的代码不降低标准，须与其他代码遵守同一套规范。
- AI 生成内容须经人类审查方可合入，不可未经审查直接提交。
- 涉及安全、认证、加密、合规的代码，由开发者主导设计，AI 辅助实现。
- AI 进行大规模变更前，须按第一章变更确认机制获得批准。

## 十、运行规则速查（面向 AI 工具）

每次对话开始或执行任务时，AI 工具需遵循：

1. **先理解，后动手** — 不清楚需求细节时先提问，不乱猜。
2. **大改动先确认** — 参考第一章变更确认机制，不擅自做架构级改动。
3. **每一步有回退方案** — 改动前用 Git 记录存档点；实验性修改在独立分支上进行。
4. **输出对齐上下文** — 阅读相关代码后再修改；变更保持与已有风格一致。
5. **修改完验证** — 跑通相关测试；新功能补充测试；破坏性变更更新文档。
6. **提交有章法** — 遵循 Conventional Commits；逻辑独立变更分次提交。
